## 0.5 v0.4 哲学（Jay 反馈后 2026-06-28 修订）

### R3 批判 R1 之前方案

R1 之前设计了 **5 支柱 30+ 决策点 15-20 天实施**——典型的 over-engineering。

**Jay 的核心洞察**（R6 风险意识的最佳体现）：
- **本质目的**：给 AI 提供**人类已审核的**好上下文
- **设计复杂 = 没人用**——简单 > 完美
- **核心约束**：
  - `.enjoyknowledge/` = **人类已审核** = 正式 KB（trust = 1.0）
  - 任务/工作流 → **放别处** = stage/（与 KB 物理分离）

### v0.4 极简原则

1. **人类是 authority anchor**——KB 里所有内容都是人类手动添加/审核
2. **简单 > 完整**——C10 信任 / C11 生命周期 / C12 同步 都是为"AI 自动维护"场景设计的，你场景不需要
3. **物理分离 > 状态字段**——stage/（任务/工作流）vs .enjoyknowledge/（KB），**不混**
4. **AGENTS.md > frontmatter 元数据**——AI 工具读 markdown 内容比读 YAML 字段更直接

### v0.4 砍掉什么（v0.5+ 再做）

| 砍掉 | 原因 | 何时做 |
|------|------|--------|
| C10 信任体系（confidence / last_verified / feedback_count / source / scope）| 你的 KB = 人类审核 = 信任 = 1.0——trust 字段是给"AI 推测"用的，你场景不需要 | v0.5+（如果未来有 AI 捕获流程）|
| C11 生命周期（4 状态机 + archive/ 物理移动）| 人类审核 = 直接 active——不需要 draft / deprecate 状态 | v0.5+（如果需要 deprecate 旧知识）|
| C12 同步（code ↔ KB 一致性检测）| KB 手动维护，code 改了人类自己更新——自动化检测价值低 | v0.5+（如果 KB 规模爆炸增长）|
| 复杂 frontmatter schema | 人类手动加知识时不想填 10 个字段——破坏心流 | v0.4 不做 |

### v0.4 保留什么（核心 3 能力）

1. **C9 capture（扩展）**——`ek capture --from-commit` 提议知识 → 人类审核后落地
2. **`.enjoyknowledge/AGENTS.md`**——AI 工具读这个文件拿到所有 KB 上下文
3. **`stage/` 目录**——任务/工作流暂存，与 KB 物理分离

**frontmatter 字段极简设计**（Jay 反馈后 2026-06-28 修订，**R6 v6 教训**）：

**R1 R3 自我批判**：R1 之前 C10 调研设计 10 字段 trust 体系 + C11 调研设计 4 状态生命周期 = **过度抽象**。你的洞察"知识文档下肯定是人类已审核的"——意味着 trust/lifecycle 字段**多余**。

**v0.4 极简 frontmatter 字段**（v0.2 已有 4 字段，**不加任何字段**）：

```yaml
---
id: utf8-windows
kind: gotcha
created: 2026-06-28
author: jay
---
```

**字段定义**（极简，**不再扩展**）：
- `id` —— 知识唯一标识（kebab-case）
- `kind` —— 知识类别（gotcha/rule/decision/architecture/business/pattern/command/context/contract/convention/template）
- `created` —— 创建时间（ISO 8601）
- `author` —— 写作者

**R1 决策**：
- **不加** trust 字段（confidence/source/last_verified/feedback_count）—— 人类审核 = trust = 1.0
- **不加** lifecycle 字段（status/deprecation_reason）—— 人类审核 = 直接 active
- **不加** scope 字段（globs/projects）—— 知识文件由人类管理，scope 走分类目录而非字段
- **不加** depends_on 字段——依赖关系通过文件引用（`[[id]]`）表达

**R6 v6 教训**（已写进 plan final）：
- 人类审核 = trust = lifecycle = scope = depends_on = **全部不需要字段**
- KB 是**人类手动管理**——字段 = **管理 metadata**（id/kind/created/author），不是**知识 metadata**
- 简单 4 字段 = **形式服从功能**——人类能写、AI 能读、足够追溯
- 之前 10 kind 子目录也**保留**——这是 v0.2 已有结构，**不再扩展**（v0.4 不加新 kind）

**删除说明**（Jay 反馈后 2026-06-28）：
- 之前 R1 设计的 `workflow/` 子目录（plans/reviews/sessions/retros）**全部删除**
- 原因：8 文件结构**完全覆盖**了 workflow/ 的所有功能
  - plan → `tasks/<task-id>/plan.md`
  - review → `tasks/<task-id>/delivery.md`（self-review 部分）
  - session log → `tasks/<task-id>/changes.md`（append-only）+ `summary.md`（stage log）
  - retro → `tasks/<task-id>/summary.md`（lessons learned）
- workflow/ 是 R1 之前调研的**过度抽象**——把"plan/review/session/retro"当独立类别，没有跟着任务走
- 实际工作流中，workflow 产物都是**任务的一部分**——它们应该跟着 task 走
- **R6 v5 教训**：调研要**跟着数据走**（实际工作流），不要**先入为主**（理论抽象）

### 1.2 核心流程

```
                              人类手动写（显式让 AI 也可）
                                    ↓
                            .enjoyknowledge/*.md  ←─── AGENTS.md ──→ AI 工具自动读
                                    ↑
                            ek promote (人类手动 / 显式 AI 操作)
                                    ↑

### 1.2.1 stage 任务 8 文件写入时机（双 AI 调研后 2026-06-28 修订）

**R1 R3 自我批判 + 双 AI 整合**：
- R1 之前 4 文件（summary + decisions + files-changed + review-checklist）**漏掉 P1 需求 + P2 设计独立文件**
- claude 推荐 5 文件（+requirements + design，decisions 合并到 design）
- codex 推荐 **8 文件**（再 + plan + changes + tests + delivery + review，summary 留 cross-stage 总览）
- **why 决策**（双 AI 整合后）：**8 文件结构**——每文件单一触发器 + 单一 owner stage

**5 阶段 × 8 文件 × 3 审批门**（行业对标 Kiro / Copilot Workspace）：

| 阶段 | 文件 | AI 行为 | 人类角色 |
|------|------|---------|---------|
| **P1 需求** | `requirements.md` | EARS 格式写需求 + ACs + 边界 | **审批者**（硬门 1） |
| **P2 设计** | `design.md` | 技术选型 + trade-off + 架构图 | **审批者**（硬门 2） |
| **P2b 计划** | `plan.md` | subtask 分解 + 执行顺序 + 回滚 | 观察者 |
| **P3 编码** | `changes.md` | 每次编辑后 append 一行 | 观察者（mid-checkpoint） |
| **P4 测试** | `tests.md` | 覆盖矩阵 + 结果 + 风险 | **审核者**（软门） |
| **P5 交付** | `delivery.md` | commit + PR draft + KB promote | **决策者**（硬门 3） |
| **跨阶段** | `summary.md` | 任务开始写目标/ACs，每阶段后追加 stage log | 必读 |
| **跨阶段** | `review.md` | AI 预填 checklist（**不勾选**），人类审核勾选 | 必填 |

**8 文件 6 个写触发动作**（每文件 1 个触发器）：

| 文件 | 首次写入 | 追加 / 重写触发 | 写者 |
|------|----------|-----------------|------|
| AGENTS.md | 仓库初始化 | 协议变更时 | 人类/AI bootstrap |
| summary.md | 任务分配后 | 每阶段完成后追加 stage log；最终完成后追加完成总结 | AI |
| requirements.md | 读 issue + 3+ 源文件后 | 人类回答 clarification | AI |
| design.md | requirements confirmed 后 | 人类提议替代方案时 | AI |
| plan.md | design confirmed 后 | 每 subtask 完成 | AI |
| changes.md | 首次 file edit 后 | **每次 file edit 都 append** | AI |
| tests.md | 首次 test run 前 | 每次 test run 后追加 Execution Results | AI |
| delivery.md | 全部测试通过后 | commit message 修改时 | AI |
| review.md | 各阶段完成后累积 | 人类勾选 [x]（**AI 不勾选**）| AI 预填 + 人类勾选 |

**关键设计**（双 AI 一致）：
- 4 文件**不是同时写**，而是**分阶段写入**
- AI 工具**在每个工作节点都触发写入**——不是"完成时一次写"
- 人类可以**随时查看进度**——而不是"完成才看"
- **append-only**：每个文件都是追加写入，保留历史（不覆盖）
- **review.md 反模式**：AI 预填但**不勾选**——只有人类审核后才勾选（保留 authority 边界）

### 1.3 AI 工具使用方式

**任何 AI 工具（Claude Code / Cursor / Continue / Kiro）启动时**：

1. 读项目根的 `AGENTS.md`（人类写的项目说明）
2. 读 `.enjoyknowledge/AGENTS.md`（KB 索引 + kind 描述 + 写入规则）
3. 读 `.enjoyknowledge_stage/AGENTS.md`（任务暂存 8 文件写入规范）
4. 读具体 kind 目录下的 .md（按需）

**写入时**：

**写 `.enjoyknowledge/`**（人类已审核的正式 KB）：
- ❌ AI 工具**不直接写**——除非人类显式说"帮我加一条 gotcha"
- ✅ 人类手动编辑，或显式让 AI 写
- ✅ promote 流程：`.enjoyknowledge_stage/drafts/` → `.enjoyknowledge/<kind>/`（人类 review 后操作）

**写 `.enjoyknowledge_stage/`**（AI 任务暂存）：
- ✅ AI 工具**随工作进行不断写入** `tasks/<task-id>/` 8 文件
- ✅ 每文件单一触发器 + 单一 owner stage
- ✅ append-only 模式：每个文件都是追加，保留历史
- ✅ workflow/ 目录存放 plan / review / session log / retro
- ✅ drafts/ 存放待 promote 的知识草稿

**绝对禁止**：
- ❌ AI 工具自动写 `.enjoyknowledge/*.md`（必须人类显式）
- ❌ AI 工具写 frontmatter 字段
- ❌ AI 工具自动 promote（必须人类操作）
- ❌ AI 工具勾选 review.md（review.md 只由人类勾选）

- 用户用得了 > 设计完整——**形式服从功能**

### 3.3 知识 frontmatter 字段设计（claude 原创）

```yaml
---
# ===== Trust 四维度 =====
confidence: 0.95              # [0.0-1.0] AI 推测=0.3-0.7, 人写=0.9+, 社区=0.7-0.9
last_verified: 2026-06-20     # ISO 8601——最后人工验证时间
expires: 2026-09-20           # 可选——硬过期日
source: human-authored        # human-authored | ai-inferred | community-shared | test-confirmed
scope:                        # Glob 模式控制生效范围
  - "src/**/*.rs"
  - "Cargo.toml"

# ===== 来源追溯 =====
author: jay@enjoyflow
git_sha: a1b2c3d              # commit hash
reviewed_by: reviewer-id      # 可选——peer review

# ===== 反馈追踪 =====
feedback_count: 3
negative_feedback: 0          # "这条不准"标记次数
last_feedback: 2026-06-25
---
```

**关键设计**：
- `feedback_count` / `negative_feedback` = C10 区别于所有对标系统的**原创**——行业没有量化反馈机制
- `test-confirmed` 免死金牌（Cursor 社区模式）—— 测试通过 = 永久 fresh
- `author + git_sha` 替代密码学签名（不重不漏）

### 3.4 降级策略（6 级规则）

```
confidence ≥ 0.85 AND last_verified < 90d ago  → 正常注入
confidence ≥ 0.85 AND last_verified ≥ 90d ago  → 注入 + [MAY BE STALE] 标记
confidence 0.50-0.85                             → 注入 + [LOW CONFIDENCE - VERIFY]
confidence < 0.50                                → 降级到"参考"区
negative_feedback ≥ 3                             → 强制降级 + doctor 警告
source == "ai-inferred" 且无 last_verified       → confidence-0.2 惩罚
source == "test-confirmed"                        → 跳过 freshness decay
expires 已过                                      → 移除注入 + 写入"已过期"清单
```

**关键策略**：
- **软降级优先于硬删除**（B 决策）—— 标记后保留比隐藏好
- **Feedback 驱动降级** —— `negative_feedback` 是 C10 核心差异化
- **`test-confirmed` 免死金牌** —— 测试是最强的新鲜度证明

### 3.5 Rust 可行性（claude 评估）

## 5. C12 知识同步（已完成双 AI 调研）

### 5.1 核心问题（双 AI 一致）

C9 解决"知识怎么流动"，C10 解决"知识可不可信"，C11 解决"知识该不该活"，**C12 解决"code ↔ KB 一致性"**。

**3 大价值**（claude）：
- **AI 不会盲信过时知识**：代码改了但 KB 没改 → AI 按旧规范生成代码 → PR 被拒
- **人类不会重复考古**：KB 改了但代码没改 → 新人按新规范 review → 发现代码"不符合"
- **知识之间不矛盾**：ADR-001 说 PostgreSQL，ADR-015 说 MongoDB——两条知识需要 cross-ref 一致性

**关键数据**（claude 引用）：
- 2026 Cortex Benchmark：AI 基于过期上下文工作时事故率上升 23.5%
- ETH Zurich ICSE 2026：LLM 生成的过时上下文文件降低任务成功率 2-3% + 增加 20%+ 成本

**codex 关键洞察**：
- 3 类失败模式（correctness / completeness / cross-consistency）
- **C12 是行业空白**——所有 7 系统都没有自动化 knowledge-code 一致性检查
- Unison 的"content-addressable"是终极方案但要求非标准语言
- Livebook 的"co-located"对人类可见但不可自动检测

### 5.2 4 同步方向详解（双 AI 整合）

### 5.3 3 类冲突 + 检测算法（双 AI 整合）

**Type 1：Reference Staleness（code 改了，KB 引用断裂）**
- claude 命名："代码→KB 滞后" / "文件路径变更"
- codex 命名："Stale reference"
- 检测：`git diff --name-only HEAD~1..HEAD` → 解析每个知识条目的 `applies_to` → 死引用 = error，移动 = warning
- 解决：自动（`git log --follow` 追踪 → 更新 `applies_to`）

**Type 2：Content Divergence（code 行为变了，KB 描述旧行为）**
- claude 命名："代码→KB 滞后（行为层）" / "API 描述错误"
- codex 命名："Content divergence"
- 检测：`source=ai-inferred` → `re-capture --dry-run` 重新生成对比 frontmatter；`source=human-authored` → embedding cosine distance > 0.7 触发 review
- 解决：建议（生成 PR，人类 review）

**Type 3：Knowledge Contradiction（两条知识矛盾）**
- claude 命名："KB↔KB 矛盾"
- codex 命名："Knowledge contradiction"
- 检测：scope 重叠 + cosine similarity > 0.8 + 对立主张 → `doctor sync --cross-ref`
- 解决：标记（promote 到 `knowledge-tasks/` 任务，priority 由 confidence 决定）

**避免误报的 3 闸门**（双 AI 共识）：
1. **scope 交集测试**——必须 scope.globs 重叠
2. **语义 diff 而非文本 diff**——空白修改不触发
3. **最小变更阈值**——单行注释编辑不触发

## 6. C13 知识暂存区（已完成双 AI 调研）

### 6.1 核心问题（双 AI 一致）

**正式 KB 是"已沉淀"**——每条知识有 confidence、status、scope，经过 C9-C12 治理。
**编码过程产生的是"未沉淀"**——plan / review / session log / drafts 80% 不需要进正式 KB，但 20% 需要在合适时机"毕业"进去。
**暂存区 = 80/20 分流的缓冲区**。

3 大理由（双 AI 共识）：
1. **低摩擦捕获**——debug 中无 frontmatter 时间，scratchpad 先收下后 formalize
2. **AI 产生中间产物不是知识**——plan / log / retro 性质 ephemeral
3. **归档决策需要时间距离**——draft 是否变永久知识需 aging out 决策窗口

### 6.2 4 暂存维度详解（双 AI 整合）

| 维度 | 触发 | 内容 | 生命周期 | 提升到 KB？ | 关键设计 |
|------|------|------|----------|-------------|----------|
| **任务暂存** | 用户 `mkdir stage/tasks/REQ-001` | free-form markdown + 决策草案 | `status: completed` → archive 触发 | ✅ `archive --req` 提取 `##` entries | 人类判断完成时机 |
| **AI workflow** | AI session 启动 | plan.md / log.md / review.md / retro.md / snapshots/ | 7d active / 30d stale / 90d hard delete | ❌ 不直接（review 可能 inspire task）| **时间清理是唯一可行策略** |
| **Scratch draft** | 手动 `draft new` | 未完成 ADR / pattern | 180d orphan warning + 手动 promote | ✅ `verify <id>` 改 status draft→active | **物理分离 drafts/ 子目录** = 防御性深度 |
| **Session cache** | AI tool call | tool stdout / 解析结果 / 中间推理 | session 结束即清 | ❌ 无 | `.gitignored` + 不可重建成本为零 |

### 6.3 7 行业系统对标（双 AI 整合）

| 系统 | 暂存机制 | 时间策略 | 提升路径 | 关键创新 | URL |
|------|----------|----------|----------|----------|-----|
| **GitHub Issues** | Issue body + comments as scratchpad; `Close` action | 手动 close + stale bot 60d | Issue → PR → KB (docs-as-code) | task 级 scratchpad | [github.com](https://github.com/) |

| **Notion** | 页面 scratchpad（无 draft/published 区分）| 手动 trash 30d | 任意 page 链接到任意 db | 通用页即 scratch | [notion.so/help](https://www.notion.so/help/) |
| **Obsidian** | Daily Notes (temporal inbox) + scratch/ 独立目录 + Knomo | daily decay by date | daily note → atomic note + [[link]] | 5-workstream 设计 | [help.obsidian.md](https://help.obsidian.md/Plugins/Daily+notes) |
| **Logseq** | Journal (scratchpad) + Pages (KB) + block-level refs | journal 不删但滚出视野 | block ref → page → linked KG | journal=inbox 模式 | [docs.logseq.com](https://docs.logseq.com/) |

**关键发现**（双 AI 共识）：**6 行业对标都分离"暂存"vs"正式 KB"**（Zettelkasten 经典模式），但**没有任何一个 4 维分得清**。GitHub/Linear 合并任务+草稿+工作流，Obsidian/Logseq/Foam 分离 daily scratch 但不管 AI workflow 产物。Cline 单体扁平 6 文件。C13 的 4 维分立是真实差异化。

### 6.4 整合设计（双 AI 整合）

**核心争议**：claude 推荐**统一 `stage/`**（PARA 原则"暂存是一个动作类别"），codex 推荐**分离 `knowledge-tasks/` + `.workflow/`**（v0.2 已实现 knowledge-tasks/）。

**整合方案**：

```
项目根/
├── .enjoyknowledge/              # 正式 KB（C9-C12 治理）
│
├── stage/                       # 统一暂存层（C13 治理）——双 AI 共识
│   ├── tasks/                   # 用户驱动（原 knowledge-tasks/）
│   │   └── REQ-001/
│   │       ├── notes.md
│   │       ├── trial-1.md
│   │       └── status.md
│   │
│   ├── workflow/                # AI 驱动（原 .workflow/）
│   │   ├── plans/               # 7d 过期
│   │   ├── reviews/             # 7d 过期
│   │   ├── sessions/            # 轮转清理
│   │   └── retros/              # 90d 保留（可 promote）
│   │
│   ├── drafts/                  # 未完成知识草稿（人类/AI 共用）
│   │   └── adr-why-anyhow.md    # status: draft
│   │
│   └── .archive/                # 已归档（同原 knowledge-tasks/.archive/）
│
└── .session/                    # 会话缓存（.gitignored，OS tmpdir 级）
    └── cache/
```

**避免重叠的 3 规则**（双 AI 共识）：
1. **方向决定目录**：用户主动创建 → `stage/tasks/`；AI 自动生成 → `stage/workflow/`
2. **内容决定 promote**：`stage/` → `.enjoyknowledge/` 走 `promote`——只有人类触发
3. **时间决定清理**：`stage/workflow/` 按时间清理，`stage/tasks/` 按状态清理

**迁移策略**（why 决策）：
- v0.2 `knowledge-tasks/` → v0.4 `stage/tasks/`（软链接 + deprecation 通知）
- archive.rs 路径常量更新为 `stage/tasks/`
- v0.4 软链接过渡 → v0.5 完全迁移

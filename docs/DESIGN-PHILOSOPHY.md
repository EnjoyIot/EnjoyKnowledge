## 一、第一性原理：LLM 与软件工程的四个根本矛盾

### 1.1 LLM 的本质属性

设计一个 AI 编程工具的知识框架，起点不是竞品分析，不是用户痛点访谈，而是**LLM 本身是什么**。

LLM 在编程场景下有六个不可回避的本质属性——三个短板、三个长处：

| | 属性 | 在编程中的含义 |
|---|---|---|
| **短板 1** | 上下文窗口有限 | AI 一次只能处理有限的信息。塞入越多，越容易"失焦"。窗口之间无自动继承 |
| **短板 2** | 概率性生成，不可靠执行 | AI 没有真正的理解和判断力。会幻觉、会遗漏约束、会在简单任务上犯错 |
| **短板 3** | 会话隔离，无持久记忆 | 每次对话是独立的。上一轮 AI 学到的经验、发现的坑、做的决策，不会自动带到下一轮 |
| **长处 1** | 意图 → 结构化输出 | AI 能将自然语言意图转化为代码、文档、配置、测试用例等结构化产物 |
| **长处 2** | 大规模模式匹配 | AI 见过的代码量和模式远超任何人类，天然知道最佳实践、常见架构、惯用写法 |
| **长处 3** | 速度与吞吐量 | AI 在数分钟内完成人类需要数小时的生成、分析、审查、重构工作 |

### 1.2 四个根本矛盾

软件工程的本质属性与 LLM 的三短板之间，产生了四个不可调和的矛盾。这四个矛盾是 EnjoyFlow 存在的**第一性理由**。

```
矛盾 1: 信息总量 > 上下文窗口
  一个中等项目: 数万行代码 + 数十份设计文档 + 数百条历史决策 + 团队规范 + 业务规则
  一个 LLM 会话: ~50K-200K tokens 有效上下文
  → 不可能把所有知识塞进一次对话。必须有一个外部系统决定"当前任务需要哪些知识"

矛盾 2: 可靠性要求 > AI 的确定性
  软件工程: 合约必须精确、行为必须一致、错误可能造成生产事故
  AI 输出: 概率性的、不保证正确的、每次可能不同的
  → 必须有不依赖 AI 判断力的不可变合约，以及独立于执行的验证机制

矛盾 3: 工程连续性 > 会话隔离性
  软件项目: 持续数月到数年，决策和经验必须跨时间传递
  AI 会话: 独立、失忆、上一轮的经验不会自动继承
  → 必须将每一次会话的产出和发现外化为持久文档，让"下一次会话"的 AI 能重新获取

矛盾 4: 团队协作复杂度 > 单 AI 线性思维
  真实团队: 多人 + 多 AI + 多工具并行工作，角色和职责动态分配
  单 AI: 线性推理，无原生协作概念
  → 必须有能力边界（谁/什么 AI 负责什么），以及协作接口（产出物如何交接）
```

### 1.3 从矛盾到框架：EnjoyFlow 的必然性

四个矛盾分别指向框架的四个核心能力：

```
矛盾 1 (信息 > 窗口)    → 精准供给能力: 按任务按场景，只给 AI 当前需要的知识
矛盾 2 (可靠性 > 确定性)  → 合约为界与风险可辨: 不可变合约 + AI 风险告知 + 独立验证 + 分级授权
矛盾 3 (连续性 > 隔离性)  → 全程留痕能力: L1 任务轨迹 + L2 决策轨迹 + L3 经验轨迹，每一层都不可跳过
矛盾 4 (复杂度 > 线性)   → 场景驱动能力: 不同任务类型有不同流程、决策关口和验证方式，不一刀切
```

这四个能力不是"好有的特性"，而是**LLM 的本质属性决定了它们必须存在**。没有任何一个现有工具完整覆盖了这四个能力——这就是 EnjoyFlow 的存在空间。

## 二、产品定位

### 2.1 定位陈述

> **EnjoyFlow 是人机共享的任务上下文层。**
>
> 它不是 AI 工具，不是知识库，不是项目管理系统。
> 它是这三者之间的**中间件**——在 AI 需要上下文时精准供给，在人需要可追溯性时提供合约，在任务结束时自动归档经验。

一句话版本：

> EnjoyFlow gives AI exactly the knowledge it needs for the current task — no more, no less.

### 2.2 EnjoyFlow 不是什么

| 不是 | 为什么 |
|---|---|
| **AI 编程工具** | 不替代 Cursor / Copilot / Claude Code / Codex。EnjoyFlow 是它们之间的上下文中间件 |
| **知识库** | 不做 Notion / Obsidian / Confluence 的替代品。EnjoyFlow 是知识库和 AI 工具之间的**供给层** |
| **通用 AI Memory** | 不做 mem0 / MemOS 的竞争者。EnjoyFlow 聚焦于**项目级、结构化的**任务知识，不做通用记忆存储 |
| **Spec 框架** | 不做 spec-kit / OpenSpec / GSD 的替代。EnjoyFlow 的合约层比 spec 更宽（含产品+设计+技术），且是规格框架的上游输入 |
| **技能/Skills 框架** | 不做 Superpowers / Agent Skills 的竞争者。EnjoyFlow 是技能执行时**拉取上下文的数据源** |
| **项目管理工具** | 不做 Jira / Linear 的替代。EnjoyFlow 从项目管理系统**读**任务，但本身不管理任务 |

### 2.3 在生态中的位置

```
         ┌──────────────┐
         │  人类开发者    │
         └──────┬───────┘
                │ 定义合约、审核产出
                ▼
┌─────────────────────────────────────┐
│           EnjoyFlow                 │
│  ┌─────────┬─────────┬──────────┐   │
│  │产品合约  │设计合约  │技术合约   │   │  ← 三层合约
│  └─────────┴─────────┴──────────┘   │
│  ┌──────────────────────────────┐   │
│  │       ContextFlow 上下文供给引擎    │   │  ← 精准供给
│  └──────────────────────────────┘   │
│  ┌──────────────────────────────┐   │
│  │       场景模板 (8+)           │   │  ← 流程编排
│  └──────────────────────────────┘   │
└────────────┬────────────────────────┘
             │ 按需供给上下文
             ▼
┌─────────────────────────────────────┐
│  AI 工具层: Cursor / Copilot /      │
│  Claude Code / Codex / ...          │
│  ┌─────────┬─────────┬──────────┐   │
│  │ Define  │  Build  │  Verify  │   │  ← 四通道
│  │(合约组装)│(执行构建)│(独立验证) │   │
│  └─────────┴─────────┴──────────┘   │
│  ┌──────────────────────────────┐   │
│  │         Learn (知识策展)      │   │
│  └──────────────────────────────┘   │
└─────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────┐
│  知识存储: 项目仓库 / Obsidian /     │
│  Notion / 文件系统 / ...            │
└─────────────────────────────────────┘
```

EnjoyFlow 是**夹在 AI 工具和知识存储之间的中间件**——它不拥有任何一端的生态位，而是让两端更好地协作。

## 三、设计哲学：五条核心原则

每一条原则都直接从一个或多个根本矛盾中推导出来，每一条原则都对应具体的结构设计决策。

### 3.1 原则一：精准供给优于全面覆盖

> 推导自矛盾 1（信息 > 窗口）

**原则陈述：** 不要在 AI 会话开始前预加载一个"完整的知识包"。在会话中，让 AI 按需、按场景主动拉取当前任务需要的知识子集。框架的职责是让"找到需要的知识"比"全部塞进去"更快、更准。

**对应的结构决策：**
- ContextFlow 是一个 **工具**（AI 主动调用），不是预生成文件（人预先准备）
- 场景模板定义每个任务类型的默认知识维度集合
- 知识文档通过 YAML frontmatter tags 实现可检索，而非依赖目录结构导航
- 上下文预算分层：持久层 < 10K tokens，任务层 < 30K tokens，会话层 < 50K tokens

**反模式（被此原则禁止的设计）：**
- 在每次 AI 启动时自动将全部知识库注入 system prompt
- 要求人在 AI 工作前手动选择和组装上下文文件
- 预生成单一的全量上下文快照文件（那会塞爆窗口）

### 3.2 原则二：合约为界，风险可辨

> 推导自矛盾 2（可靠性 > 确定性）

**原则陈述：** 约束 AI 行为的合约（产品需求、设计规范、技术架构、代码标准）是 AI 执行的稳定基准——AI 不能单方面修改。但"合约为界"不等于"无脑执行"。AI 必须忠实执行合约，同时在发现合约可能引发的风险时主动告知人类。执行归执行，开口归开口。

**框架不管理合约生命周期。** 敏捷团队通过自己的流程（sprint planning、设计评审、技术讨论）产出合约——User Story、Figma 设计稿、技术决策记录。框架只做三件事：读取合约快照、感知版本变化、在 AI 下次拉取时自动供给最新版本。合约变更是团队的事，框架只保证 AI 不会在过期快照上盲跑。

AI 产出的正确性不能由同一个 AI 在同一个会话中判断，验证必须在独立上下文（独立 AI 会话，甚至不同 AI 模型）中进行。

**AI 面对合约时的行为契约：**

| 场景 | AI 的行为 |
|---|---|
| 合约合理，执行顺利 | 照常执行 |
| 合约合理，但执行路径有已知风险 | **执行 + 告知**："按合约使用 Redis 做缓存，但这个查询模式可能导致缓存雪崩，建议加互斥锁" |
| 合约方案可行但存在优化空间 | **执行 + 建议**："按合约用 REST API，考虑到调用频率，建议后续评估 WebSocket" |
| 合约明显有误（如引用不存在的模块） | **暂停 + 升级**："合约引用的 payment-v2 模块不存在，无法继续，请确认" |
| 合约未覆盖当前场景（AI 需自行判断） | **按风险分级处理**（见下方） |

**决策权的风险分级：**

"人类决策主权"是一个危险的简化——在某些技术领域 AI 比人类更专业，而在业务判断上人类比 AI 更可靠。决策权不应由"谁是人谁是 AI"决定，而应由**决策后果的不可逆性**决定：

```
高风险（不可逆或极难回滚）
├─ 架构选型、数据模型设计、API 合约定义、产品方向
├─ AI 提案 → 人类审批 → 执行
└─ 理由：错了很难改，需要业务判断力和长期视角

中风险（可逆但有代价）
├─ 库版本选择、重构方案、错误处理策略
├─ AI 提案 → 执行 → 人类抽查
└─ 理由：AI 见过更多模式，但需要人对项目上下文把关

低风险（完全可逆，AI 最擅长）
├─ 代码格式、变量命名、测试用例结构
├─ AI 自主决策 → 事后抽查
└─ 理由：错了容易改，AI 在此类任务的准确率最高
```

**AI 的主动升级义务：**

即使在被授权自主决策的范围内，AI 也必须在以下情况主动升级给人类：
- 对提案的置信度低于阈值
- 遇到知识库中未覆盖的领域
- 多个可行方案之间无法判断（缺少业务上下文）
- 检测到修改可能影响关键模块（支付、认证、数据完整性）

**对应的结构决策：**
- 三层合约（产品/设计/技术）的当前快照存入知识库（A6/A7/A1-5+A10），由团队现有流程维护
- 合约变更时，ContextFlow 下次拉取自动拿到新版本；AI 感知到合约差异时主动向人类确认
- 合约没有"框架级锁定"——团队通过自己的流程（PR review、设计评审、sprint planning）决定合约是否就绪
- 四通道分离：Build 通道和 Verify 通道必须是不同的 AI 会话
- 场景模板的 `architecture_decision` 场景强制人类决策关口——高风险决策必须人批
- 场景模板的状态机中嵌入风险告知节点：AI 发现风险时写入任务日志，不阻塞执行（除非是致命问题）
- 验证产出（测试报告、审查结论）写入知识库，成为下一次任务的知识输入
- 知识维度 D2（Session Logs）和 D7（Failure Modes）记录 AI 的风险告知历史，供后续追溯

**反模式：**
- AI 在同一个会话中实现功能然后自我审查
- AI 发现合约问题但沉默执行——"无脑服从"
- AI 觉得合约不够好就自作主张修改——"越权执行"
- 架构文档被 AI 在执行任务时随意修改
- 验证依赖"AI 说它做对了"而不是可复现的检查
- 要求人类在所有技术决策上都审批——"过度管控"，导致瓶颈

### 3.3 原则三：人机共读，双向维护

> 推导自长处 1（意图 → 结构化输出）和长处 3（速度）

**原则陈述：** 所有知识文档必须同时对人可读（Markdown 正文）和对 AI 可解析（YAML frontmatter 元数据）。人写核心判断和设计意图，AI 做格式化、补全、关联和日常维护。一条经验只记录一次，但可以被多个场景、多个 AI 工具以不同方式消费。

**对应的结构决策：**
- 所有知识文档采用 `YAML frontmatter + Markdown body` 格式（对齐 Agent Skills SKILL.md 和 DESIGN.md 标准）
- 标签体系（tags）是 YAML 中的结构化字段，人写语义标签，AI 自动补充分类标签
- AI 生成初稿 → 人审核 → 锁定的维护循环
- ContextFlow 的 `enjoyflow_record` 工具让 AI 在执行任务期间主动记录发现，不用等人事后补

**反模式：**
- 纯人类维护的知识库（太多工作量，必烂尾）
- 纯 AI 自动生成的知识库（没有人的判断，变成无意义文本堆砌）
- 知识只为 AI 消费而写（失去了人对知识质量的审查能力）

### 3.4 原则四：全程留痕，可追溯

> 推导自矛盾 3（连续性 > 隔离性）

**原则陈述：** AI 的每一次交互——用了什么上下文、做了什么决策、产出了什么、发现了什么——都必须在知识库中留下痕迹。"留痕"不只是为了经验传承，更是为了验证可追溯（Verify 通道需要知道 Build 通道做了什么）和决策可回溯（为什么当时的方案是 A 而不是 B）。

"经验外化"（AI 发现的坑和教训）是"留痕"的一个子类——它是最容易被忽视的一类留痕，但不是留痕的全部。

**留痕的三个层次：**

```
L1: 任务轨迹（自动留痕，无需 AI 额外动作）
├─ 任务开始/结束时间、使用的场景模板、拉取的上下文维度
├─ 产出的文件清单、修改的代码范围
└─ 对应维度: D1 Task Progress、D2 Session Logs

L2: 决策轨迹（AI 主动记录）
├─ 技术取舍及原因、方案选择的 trade-off
├─ 风险告知记录（AI 在原则二中"开口"的内容）
└─ 对应维度: A8 ADR、C3 Decision Archive、D3 Decision History

L3: 经验轨迹（AI 主动记录，价值最高）
├─ 新发现的坑、模块的真实行为与文档的差异
├─ 失败的尝试、为什么失败
└─ 对应维度: C1 Gotchas、C5 Known Issues、D7 Failure Modes
```

**L1 自动完成，L2/L3 由 AI 在 `enjoyflow_record` 中主动记录。** 框架的职责是保证：如果 L2/L3 未完成，任务不算结束。

**对应的结构决策：**
- 所有场景模板的最后一个状态都是 Knowledge Capture / Archive
- L1 轨迹由框架自动生成，不依赖 AI 或人类操作
- `enjoyflow_record` 工具是 ContextFlow 工具集的核心组成部分（与 `enjoyflow_context` 平级）
- 场景模板的 state machine 保证：L2/L3 归档未完成 = 任务未完成
- D 类知识维度（D1-D7）承载所有三个层次的留痕数据

**反模式：**
- AI 在会话中发现了问题但"口头"告知用户而不写入文档（口头告知不留痕）
- 知识记录作为可选步骤放在流程末尾
- 依赖人在 AI 工作完成后手动整理经验（人不会做的）
- 只记录"发生了什么"不记录"为什么这么选"（L1 有了，L2 缺失）
- 框架只追踪代码变更，不追踪 AI 决策路径

### 3.5 原则五：场景驱动，而非阶段驱动

> 推导自矛盾 4（复杂度 > 线性）和长处 2（模式匹配）

**原则陈述：** 软件开发不是一个线性的瀑布流程。不同任务类型（新功能、修 Bug、重构、热修复、代码审查……）需要的知识维度不同、决策关口不同、验证方式不同。框架不应该预设一个"所有任务都按同一套阶段走"的流程，而应该为每种任务类型提供最适合它的流程模板。

**对应的结构决策：**
- 8 个场景模板（new_feature、bug_fix、refactor、hotfix、architecture_decision、release_deployment、code_review、monitoring_response），各有独立的状态机
- 每个场景模板声明自己需要的知识维度集合（ContextFlow preset）
- 每个场景模板有自己的分支逻辑（如 bug_fix 区分前端/数据/性能 Bug）
- 场景模板可通过 `extends` 继承和扩展，用户可自定义
- 场景模板不替代人类的流程判断——它是一个默认推荐，人可以覆盖

**反模式：**
- "所有任务都经过 需求→设计→开发→测试→部署 五个阶段"的一刀切流程
- 框架硬编码流程阶段，人无法调整
- 不区分 Bug 修复和新功能开发的上下文需求

## 四、从哲学到结构

### 4.1 三层合约模型

三层合约是从原则二（合约为界，风险可辨）和原则三（人机共读）推导出的结构。

**框架不管理合约生命周期。** 敏捷团队已有的流程产出合约内容：sprint planning 产出的 User Story 和验收条件（产品合约），设计评审产出的 Figma 设计稿（设计合约），技术讨论产出的架构决策和技术规范（技术合约）。框架的三个角色：读取快照、感知变更、按需供给。

```
产品合约 (Product Contract)
├─ 回答: "解决什么问题，给谁用，怎么算成功"
├─ 内容: User Story、验收条件、产品指标
├─ 团队维护方式: Sprint planning、PRD review、需求澄清会议
├─ 对应的知识维度: A6 Product Requirements
└─ 典型消费者: new_feature 场景、bug_fix 场景（验收条件部分）

设计合约 (Design Contract)
├─ 回答: "用户看到什么，如何交互，长什么样"
├─ 内容: UI 原型、交互规范、设计系统引用、可访问性要求
├─ 团队维护方式: 设计评审、设计验收、设计系统更新
├─ 对应的知识维度: A7 UI/UX Design
└─ 典型消费者: new_feature 场景的 pure_frontend 分支、code_review 场景（设计还原验证）

技术合约 (Technical Contract)
├─ 回答: "怎么建，用什么建，边界在哪"
├─ 内容: 系统架构、API 合约、数据模型、代码规范、技术栈决策
├─ 团队维护方式: 技术评审、ADR 流程、代码规范更新
├─ 对应的知识维度: A1-A5, A10
└─ 典型消费者: 所有场景
```

三层合约之间的关系：
- **团队自有流程维护：** 每层合约通过团队已有的敏捷流程（sprint planning、设计评审、技术讨论）更新，框架不做审批
- **引用而非包含：** 技术合约可以引用产品合约中的验收条件（如 API 必须满足某条业务规则），但不在技术合约中复制产品合约的内容
- **按需组合：** ContextFlow 根据场景模板决定拉取哪些合约层。纯后端重构可能不需要设计合约，UI 变更可能需要全部三层
- **变更感知：** 当团队更新合约后，ContextFlow 下次拉取时自动拿到最新快照。AI 发现合约差异时主动向人类确认

### 4.2 四通道执行模型

四通道是从原则二（合约为界，风险可辨）和原则四（全程留痕）推导出的执行结构。

```
┌──────────────────────────────────────────────────┐
│                  Define 通道                      │
│  ┌─────────┐  ┌─────────┐  ┌──────────┐         │
│  │产品合约  │  │设计合约  │  │技术合约   │         │
│  └────┬────┘  └────┬────┘  └────┬─────┘         │
│       └────────────┼────────────┘                │
│                    ▼                             │
│         ContextFlow (enjoyflow_context)           │
│         (按场景组装约束上下文)                      │
│         + 上下文就绪检查: 哪些合约存在、哪些缺失      │
│  角色: 产品/设计/架构 —— 定义"做什么"的规则        │
│  关键约束: 合约由团队流程维护，Define 通道只读取和组装  │
│  就绪检查: AI 列出'已就绪'和'缺失'的上下文，让人决定  │
│           是否继续，不阻塞执行                        │
└──────────────────────┬───────────────────────────┘
                       │ 约束上下文
                       ▼
┌──────────────────────────────────────────────────┐
│                  Build 通道                       │
│  AI 在约束上下文中生成代码/文档/配置                 │
│  ├─ 忠实执行合约，不单方面修改                       │
│  ├─ 发现风险主动告知（不阻塞执行，除非致命问题）       │
│  └─ 在授权范围内做低风险决策，超出范围主动升级         │
│  角色: 执行 —— 有判断力的执行者，不是无脑服从         │
│  关键约束: 不能修改合约，不能自己验证自己              │
└──────────────────────┬───────────────────────────┘
                       │ 产出物
                       ▼
┌──────────────────────────────────────────────────┐
│                  Verify 通道                      │
│  独立 AI 会话检查产出是否满足三层合约               │
│  ├─ 产品验收: 是否满足验收条件？                    │
│  ├─ 设计还原: 是否匹配设计规范？                    │
│  └─ 技术合规: 是否违反架构/规范/API 合约？          │
│  角色: 验证 —— 独立判断"做得对不对"                 │
│  关键约束: 必须与 Build 通道是不同 AI 会话/不同模型   │
└──────────────────────┬───────────────────────────┘
                       │ 验证结论 + 发现的问题
                       ▼
┌──────────────────────────────────────────────────┐
│                  Learn 通道                       │
│  将新发现归档回知识库（原则四：全程留痕）:              │
│  ├─ L1 任务轨迹: 框架自动记录（D1 Task Progress, D2 Session Logs）│
│  ├─ L2 决策轨迹: AI 记录的技术取舍和风险告知（A8 ADR, D3 Decision History）│
│  └─ L3 经验轨迹: AI 发现的坑和教训（C1 Gotchas, C5 Known Issues, D7 Failure Modes）│
│  角色: 策展 —— 保证"这次学到的下次还能用"           │
│  关键约束: L2/L3 未归档 = 任务未完成               │
└──────────────────────────────────────────────────┘
```

**与旧模型的对比：**

| | 旧（四槽位 v1） | 新（四通道） |
|---|---|---|
| 第一槽 | `architecture_contract` — 仅技术架构 | **Define** — 产品+设计+技术三层合约 |
| 第二槽 | `execution` | **Build** — 不改名，但明确受三层合约约束 |
| 第三槽 | `independent_verification` | **Verify** — 增加设计还原验证、产品验收检查 |
| 第四槽 | `knowledge_routing` | **Learn** — 改名强调"归档动作"而非"路由存储" |

### 4.3 知识分类法（ABCD 修订）

知识分类法需要反映三层合约模型。修订如下：

**A 类 — 项目知识（Project Knowledge）:**
保持 12 类，但调整 A6/A7 的定位：

| ID | 类名 | 说明 | 变更 |
|---|---|---|---|
| A1 | Architecture | 系统架构、模块划分、技术选型 | 不变 |
| A2 | Code Standards | 代码规范、命名约定 | 不变 |
| A3 | API Contract | API 契约定义 | 不变 |
| A4 | Data Model | 数据模型、Schema | 不变 |
| A5 | API Specs | 接口文档（OpenAPI 等） | 不变 |
| **A6** | **Product Requirements** | PRD、用户故事、验收条件 | **扩展：从 "Requirements" 重命名，明确为产品需求** |
| **A7** | **UI/UX Design** | 原型、交互规范、设计系统、可访问性 | **扩展：从 "Design Specs" 重命名，明确包含 UI/UX** |
| A8 | ADRs | 架构决策记录 | 不变 |
| A9 | Test Reports | 测试报告 | 不变 |
| A10 | Dependencies | 依赖清单 | 不变 |
| A11 | Environments | 环境配置 | 不变 |
| A12 | Releases | 发布记录 | 不变 |

**三层合约与知识维度的映射：**

```
产品合约 → A6  Product Requirements
设计合约 → A7  UI/UX Design
技术合约 → A1  Architecture
          + A2  Code Standards
          + A3  API Contract
          + A4  Data Model
          + A5  API Specs
          + A10 Dependencies
```

产品合约和设计合约各占一个知识维度（A6、A7），技术合约由多个技术维度组成。这不是不对称——产品需求和设计规范通常是单一文档（或少量文档），而技术架构天然分散在多个维度中。

### 4.4 场景模板对三层合约的消费

不同场景拉取不同的合约组合：

| 场景 | 产品合约 | 设计合约 | 技术合约 | 其他维度 |
|---|---|---|---|---|
| new_feature | ✅ 验收条件 | ✅ pure_frontend 分支 | ✅ 全部 | C1 Gotchas |
| bug_fix | ✅ 验收条件 | — | ✅ 相关模块 | C1 + C5 Known Issues |
| refactor | — | — | ✅ 全部 | A9 Test Reports |
| hotfix | — | — | ✅ 相关模块 | C5 + D7 |
| architecture_decision | — | — | ✅ 现状 | A8 历史 ADR |
| release_deployment | — | — | ✅ 环境配置 | C8 Deploy Checklist |
| code_review | — | ✅ 设计还原 | ✅ 全部 | C7 Review Checklist |
| monitoring_response | — | — | ✅ 相关模块 | C5 + D7 |

**场景模板不包含人类决策关口。** 框架不做流程审批——那是项目管理工具的职责。场景模板只声明每个场景需要哪些知识维度。执行时，ContextFlow 做**上下文就绪检查**：报告哪些合约存在、哪些缺失，让人决定是继续还是等待。`architecture_decision` 保留为独立场景不是因为它有"关口"，而是因为它产出的 ADR 是其他所有场景的关键上下文输入。

## 五、哲学一致性检查

每条设计决策是否忠于五条核心原则：

| 设计决策 | 原则一精准供给 | 原则二合约为界 | 原则三人机共读 | 原则四全程留痕 | 原则五场景驱动 |
|---|---|---|---|---|---|
| 三层独立合约 | — | ✅ AI 不能单方面修改合约 | ✅ Markdown+YAML | — | ✅ 按场景组合 |
| 四通道分离 | — | ✅ Build≠Verify，风险告知不阻塞执行 | — | ✅ Learn 是终态，L1 自动留痕 | — |
| ContextFlow 作为工具 | ✅ 按需拉取 | — | — | — | ✅ 场景 preset |
| 场景模板 | ✅ 每场景不同维度 | ✅ 高风险决策关口 + 风险告知节点 | — | ✅ L2/L3 归档 = 任务完成条件 | ✅ 核心模式 |
| enjoyflow_record | — | ✅ AI 发现风险时开口记录 | ✅ AI 可主动记录 | ✅ 会话中记录 | — |
| YFM + Markdown | — | — | ✅ 核心格式 | — | — |
| 风险分级授权 | — | ✅ 按不可逆性决定审批级别 | — | — | — |
